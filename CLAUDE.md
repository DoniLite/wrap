# CLAUDE.md — `@donilite/wrap` monorepo

Guide for Claude Code (or any LLM assistant) working in this repository. This is the framework's *own* source repo — not a project built with it. If you're looking for the guide meant for apps generated by `create-wrap`, that's a separate file: `packages/create-wrap/template/CLAUDE.md` (and its per-profile overrides in `packages/create-wrap/profiles/*/files/CLAUDE.md`), which gets copied into every scaffolded project.

## What this repo is

A Bun-first, Hono + Drizzle OOP backend framework, plus its project generator, as two published packages released in lockstep from the same git tag:

- **`packages/wrap`** → `@donilite/wrap` — the framework itself.
- **`packages/create-wrap`** → `@donilite/create-wrap` — the `bunx create-wrap` scaffolding CLI, which copies `packages/create-wrap/template/` (optionally diffed by a profile in `packages/create-wrap/profiles/`) into a new project.

Full API reference lives in `/docs` (Jekyll site, deployed separately) — this file is scope/orientation, not a copy of that reference.

## Architecture, in one pass

- **`Wrap`** (`packages/wrap/src/wrap.ts`) is the composition root — owns the Hono instance, global middleware (requestId, secureHeaders, cors, bodyLimit, request logging), error/404 handling, controller registration, and Swagger. `new Wrap()` → `.with(auth)` → `.register(Controller)` → `.swagger({...})` → `.listen(port)`. `.get/.post/.put/.patch/.delete/.use` cover ad-hoc routes without a full controller; `.raw` is the Hono escape hatch for anything lower-level (websocket upgrade, a custom `Bun.serve`).
- **Vertical-slice features**: each feature owns its `entity/`, `repository/`, `services/`, `web/` (controller), `DTO/`. The Drizzle table (built via `Entity(...)`, `packages/wrap/src/entity.ts`) is the single source of truth — DTOs, validation, and OpenAPI schemas are all derived from it.
- **Controller → Service → Repository**, each with a lighter base for when the full CRUD shape doesn't fit:
  - `RouterController` (no service) vs `BaseController<Service>` (CRUD, service required) — `packages/wrap/src/base.controller.ts`.
  - `WrapService` (no repository) vs `BaseService<Repo>` (CRUD, repository required) — `packages/wrap/src/base.service.ts`.
  - `BaseRepository<Tb>` — `packages/wrap/src/base.repository.ts` — is the only layer that requires Drizzle; a Drizzle table's `BaseRow` columns (`id`, `createdAt`, `updatedAt`, `deletedAt`) are what make CRUD, soft-delete, entity-scope caching, and offline sync (`findChangedSince`/`applyBatch`) all work generically.
- **Controllers compose controllers, parent → children**: `this.register(ChildController)` inside a controller's constructor mounts a child at its own `@Controller({ basePath })`, optionally with a scoped `prefix`/`middlewares`. `Wrap.register()` is the same primitive at the top level. **Route registration is lazy** (triggered on first `getApp()` call, not in the constructor) specifically so a controller's own routes always land in Hono's router *after* every child it registers — see the long comment in `base.controller.ts`'s `RouterController` constructor if touching this; it fixes a real, previously-shipped bug where a parent's own `:id`-style route silently swallowed every request meant for a child at a static prefix.
- **`AuthController`** (`packages/wrap/src/middleware/auth/auth.controller.ts`) is paradigm-agnostic: subclasses implement `authenticate()`/`revoke()`/`openApiSecurityScheme()` (all abstract, all mandatory); `authMiddleware` and the generic `guard(predicate)` authorization primitive are implemented once, generically — the framework never assumes RBAC. `JwtCookieAuthController` is the shipped reference preset. `AuthController.combine(...)` chains several as a fallback (e.g. cookie sessions for browsers, an API-key controller for everything else).
- **Registry-driven typing** (`packages/wrap/src/registry.ts`): apps augment `WrapRegistry` via declaration merging (`declare module "@donilite/wrap" { interface WrapRegistry { schema, variables, roles, identity } }`) to get typed `db.query.*`/populate, typed `c.get()`/`c.set()`, typed role checks, and a typed `c.get("identity")` — all four are optional and independent; an app can adopt whichever it needs.
- **Swagger** (`packages/wrap/src/swagger/index.ts`) is generated entirely from decorator metadata + the registered `AuthController` — no separate OpenAPI authoring. A controller's real mount path is resolved by walking the `register()` chain recorded in `decorators/registries.ts` (`resolveControllerPath`), not just its own `basePath` — this matters if you're debugging a path showing up wrong in `/docs`.

## Repo conventions (load-bearing — don't skip these)

- **Commits**: small, atomic, one concern each. Imperative subject line. Body explains *why*, not *what* — the diff already shows what changed. Every commit ends with:

  ```
  Co-Authored-By: Claude Sonnet 5 <noreply@anthropic.com>
  ```

  Only commit when explicitly asked. Never `--amend` published commits; create a new commit instead.
- **Verification, every time, before considering anything done** (both packages):

  ```bash
  cd packages/wrap && bun run build && bun run typecheck && bun run lint
  cd packages/create-wrap/template && bun run typecheck && bunx eslint . && bun test
  ```

  `packages/wrap` ships **zero internal tests** — the ~55 tests in `packages/create-wrap/template/tests/` (Wrap composition, AuthController, sync repository methods, swagger generation, `register()` parent→child composition, ...) are the framework's actual coverage, exercised through the template. Never let that suite shrink; it's the only thing standing between a framework change and a silent regression. `packages/create-wrap/src/*.ts` (the CLI itself) has no dedicated tsconfig/lint/typecheck in this repo — verify it with `bun build --target=bun ./src/index.ts --outdir=/tmp/...` (catches syntax/import errors) and, when the change is behavioral, an actual scaffold + `bun install`/typecheck/lint/test of the generated output.
- **No comments explaining *what* the code does** — names should already do that. A comment earns its place only when it explains a non-obvious *why*: a hidden constraint, a workaround, a subtle invariant, something that would surprise the next reader. This repo's existing comments are a good calibration reference (e.g. the ordering comment in `base.controller.ts`, the `Symbol.for` reasoning in `auth.controller.ts`).
- **Empirically verify claims about runtime/library behavior before asserting them** — this session's history is full of exactly this pattern: writing a tiny throwaway script under `/tmp` (or the scratchpad), running it, deleting it, *then* stating the conclusion. Applies especially to Hono routing precedence, Node/Bun stdin/TTY quirks, and anything about how a third-party library actually behaves versus how its docs describe it.
- **Destructive/shared-state actions need explicit confirmation first**: force-push, tag deletion/recreation, `git reset --hard`, merging to `main`, publishing. Reversible local actions (edits, local commits, local branches) don't.
- **Don't design for hypothetical future requirements.** No abstractions, feature flags, or generalized configs for a need that doesn't exist yet in this codebase.

## The `create-wrap` profile system

`packages/create-wrap/template/` **is** the full-backend profile (Postgres + Drizzle, Redis cache, realtime, auth — everything). Every other profile is a *diff* against it, not a separate template tree: `packages/create-wrap/profiles/<id>/remove.txt` lists paths to delete from the copied base, `packages/create-wrap/profiles/<id>/files/` layers files on top (new or overriding). See `packages/create-wrap/profiles/README.md` for the exact mechanics before adding or modifying a profile. Current profiles: `full-backend`, `lightweight-api`, `api-aggregator`, `fullstack-ssr`, `gateway`.

## Release

Tag-driven (`.github/workflows/release.yml`): `git tag vX.Y.Z && git push origin vX.Y.Z` builds, tests, sets both packages' `package.json` version to the tag (via `npm pkg set`, which is idempotent — safe whether the version was already bumped in the tagged commit or not), publishes both to npm via Trusted Publishing (OIDC, no token), then creates a GitHub Release with auto-generated notes. If a tagged release fails, fix forward and either re-tag (delete + recreate, confirm with the user first) or bump to a new patch tag — never force-push `main`.
